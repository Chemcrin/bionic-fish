# 仿生鱼 ESP32-S3 与 STM32F103C8T6 通信方案

> 本文档整理当前已商定的通信架构、硬件准备、数据协议、上位机控制方式、ESP32 与 STM32 串口互联方式，以及软件层面的可靠性设计。当前目标不是直接生成可烧录代码，而是形成后续开发固件、网页控制台和调试工具时可遵循的技术方案。

---

## 1. 总体结论

| 项目 | 已商定方案 |
|---|---|
| 总体架构 | 上位机通过 WiFi 连接 ESP32-S3，ESP32-S3 通过 UART 连接 STM32F103C8T6 |
| ESP32-S3 角色 | 无线通信网关、网页服务器、WebSocket 服务器、协议转换器 |
| STM32F103C8T6 角色 | 舵机控制、传感器采集、运动控制、安全保护 |
| 上位机方式 | 优先不写原生 App，使用浏览器访问 ESP32 内置网页 |
| 无线方式 | 优先采用 WiFi AP 模式，BLE 可作为备用配置或低频控制方式 |
| 上位机数据格式 | JSON over WebSocket，易调试、易扩展、跨平台 |
| ESP32 与 STM32 数据格式 | 初期可用 JSON 或简化帧；正式建议使用二进制 UART 帧 |
| 控制原则 | 上位机发送“运动意图”，不要直接发送每个舵机角度 |
| 可靠性机制 | seq 序号、CRC、ACK、心跳、超时停机、参数限幅、看门狗 |

推荐通信链路：

```text
Windows / Android / HarmonyOS 浏览器
              ↓
        WiFi 2.4GHz AP
              ↓
ESP32-S3：网页服务器 + WebSocket + JSON解析 + UART协议转换
              ↓
        UART 串口
              ↓
STM32F103C8T6：舵机控制 + 传感器采集 + 运动控制 + 安全保护
```

## 2. 系统分层设计

| 层级 | 连接对象 | 推荐协议 | 数据格式 | 主要内容 |
|---|---|---|---|---|
| 上位机 ↔ ESP32-S3 | Windows、Android、HarmonyOS、平板、手机 | WiFi AP + WebSocket | JSON | 控制命令、参数配置、状态显示 |
| ESP32-S3 ↔ STM32F103C8T6 | 两块芯片内部通信 | UART | 二进制帧，初期可 JSON | 舵机指令、传感器数据、执行反馈 |
| STM32 ↔ 执行机构/传感器 | 舵机、电机、IMU、深度传感器 | PWM、I2C、ADC、GPIO | 驱动层数据 | 实时控制、姿态反馈、安全检测 |

核心分工：

| 模块 | 不建议承担的任务 | 建议承担的任务 |
|---|---|---|
| ESP32-S3 | 不直接做复杂实时舵机控制 | 无线通信、网页控制台、WebSocket、JSON解析、UART转发 |
| STM32F103C8T6 | 不直接处理 WiFi/网页/复杂网络通信 | 舵机 PWM、电机 PWM、传感器采集、控制算法、安全停机 |
| 上位机 | 不直接控制每个舵机的瞬时角度 | 发送前进、转向、上浮、下潜、模式切换等高层控制量 |

## 3. 硬件准备与接线

### 3.1 ESP32-S3 与 STM32 串口连接

ESP32-S3 和 STM32F103C8T6 都是 3.3V TTL 串口电平，正常情况下可以直接连接。

| 连接项 | 接法 |
|---|---|
| ESP32-S3 TX | 接 STM32 RX |
| ESP32-S3 RX | 接 STM32 TX |
| ESP32-S3 GND | 接 STM32 GND |
| 逻辑电平 | 3.3V TTL |
| 推荐波特率 | 初期 115200，稳定后可考虑 460800 或 921600 |

示意：

```text
ESP32-S3 TX  ---->  STM32 RX
ESP32-S3 RX  <----  STM32 TX
ESP32-S3 GND ----  STM32 GND
```

注意：这里使用的是 TTL UART，不是 RS232 电平，不能直接接 DB9/RS232 那类正负电平接口。

### 3.2 电源与抗干扰

| 项目 | 建议 |
|---|---|
| ESP32-S3 供电 | 使用稳定 3.3V 电源，电流余量建议 500mA 以上 |
| STM32 供电 | 独立稳压，注意与 ESP32 共地 |
| 舵机供电 | 不要从 STM32 或 ESP32 取电，应使用独立舵机电源 |
| 电机供电 | 与逻辑电源分开，电机驱动端预留足够电流余量 |
| 共地 | ESP32、STM32、舵机电源、电机驱动必须共地 |
| 去耦 | ESP32、STM32、电机驱动、舵机电源附近加电容 |
| 调试接口 | 两块板都建议保留独立 USB-UART 调试口 |

### 3.3 外接 2.4GHz 天线注意事项

你使用的是带天线基座的 ESP32-S3，并延伸出约半米长的 2.4GHz 天线/馈线，因此 WiFi AP 方案是优先推荐的。

| 项目 | 注意事项 |
|---|---|
| WiFi AP 适配性 | 外接天线有利于扩大通信范围，适合 ESP32 自建热点 |
| 水下衰减 | 2.4GHz 在水中衰减严重，天线最好尽量露出水面或靠近水面 |
| 馈线长度 | 半米馈线可用，但馈线越长损耗越大，应使用合格 50Ω 同轴线 |
| 天线位置 | 远离电机、电池、大电流线、金属和碳纤维结构 |
| 防水 | IPEX/SMA 接头、馈线出口、天线固定处必须密封 |
| 天线匹配 | 不建议随意用普通导线替代 2.4GHz 天线 |

## 4. 上位机与 ESP32 的通信方案

### 4.1 推荐方式：WiFi AP + 浏览器网页 + WebSocket

ESP32-S3 自己创建 WiFi 热点，上位机连接该热点后，通过浏览器访问 ESP32 内置网页。

示例配置：

```text
SSID: BionicFish_ESP32
Password: 12345678
IP: 192.168.4.1
Web: http://192.168.4.1
WebSocket: ws://192.168.4.1/ws
```

| 优点 | 说明 |
|---|---|
| 跨平台 | Windows、Android、HarmonyOS 都可以用浏览器访问 |
| 不依赖路由器 | 适合户外、水池、比赛场地 |
| 开发成本低 | 不需要一开始开发 Android/HarmonyOS/Windows 原生 App |
| 交互方便 | 网页可提供摇杆、按钮、滑条、状态显示 |
| 实时性较好 | WebSocket 适合 10Hz 到 50Hz 级控制数据 |

访问流程：

```text
1. ESP32-S3 上电后开启 WiFi AP
2. 手机/电脑连接 BionicFish_ESP32 热点
3. 浏览器访问 http://192.168.4.1
4. 网页加载控制界面
5. 网页通过 WebSocket 连接 ws://192.168.4.1/ws
6. 网页周期性发送控制 JSON
7. ESP32 解析 JSON 后转成 UART 指令发给 STM32
8. STM32 执行动作并回传状态
9. ESP32 将状态通过 WebSocket 推送回网页
```

### 4.2 备选方式对比

| 方式 | 是否推荐 | 适合场景 | 主要问题 |
|---|---|---|---|
| WiFi AP + WebSocket | 强烈推荐 | 初期开发、比赛、户外遥控 | 手机连接热点后可能无法同时上网 |
| WiFi STA | 可选 | 实验室、接入路由器、局域网调试 | 依赖外部路由器和配网 |
| BLE | 可作为备用 | 参数配置、低频控制、应急停止 | 实时控制体验通常不如 WiFi |
| UDP | 后期可选 | 低延迟遥控 | 需要自行处理丢包和乱序 |
| 原生 App | 后期可选 | 产品化、复杂 UI | Windows/Android/HarmonyOS 需要分别开发 |

结论：第一阶段优先使用 **WiFi AP + 内置网页 + WebSocket + JSON**。

## 5. 是否需要写 App 或软件

| 方案 | 是否需要用户安装 App | 是否需要开发 | 推荐阶段 |
|---|---|---|---|
| ESP32 内置网页 | 不需要 | 需要写 ESP32 固件和网页文件 | 第一阶段推荐 |
| Windows Python/C#/Qt 上位机 | 需要电脑运行软件 | 需要开发 PC 软件 | 调试阶段可选 |
| Android/HarmonyOS 原生 App | 需要安装 App | 需要移动端开发 | 后期优化 |
| BLE 调试工具 | 可用现成工具 | ESP32 侧需要写 BLE 服务 | 参数调试可选 |

需要明确一点：ESP32 不会自动提供仿生鱼控制网页。需要在 ESP32 程序中实现：

| ESP32 功能 | 用途 |
|---|---|
| WiFi AP | 创建热点 |
| WebServer | 提供 HTML/CSS/JavaScript 控制页面 |
| WebSocket Server | 与网页实时收发控制数据 |
| JSON 解析 | 解析网页发来的控制命令 |
| UART 通信 | 将控制命令转发给 STM32，并接收 STM32 状态 |
| 安全逻辑 | 连接超时、失联停机、参数限幅、错误回报 |

## 6. 上位机发给 ESP32 的 JSON 数据帧

### 6.1 为什么先用 JSON

| 优点 | 说明 |
|---|---|
| 易读 | 调试时可以直接看懂字段含义 |
| 易改 | 后续新增字段方便 |
| 跨平台 | JavaScript、Python、C#、Android 都容易处理 |
| 适合网页 | 浏览器天然支持 JSON |
| 开发速度快 | 比二进制协议更适合早期调试 |

缺点是比二进制帧更占带宽，但对 WiFi 控制仿生鱼来说完全可以接受。即使每秒发送 20 次，每条 JSON 约几十到一百多字节，WiFi 压力也很小。

### 6.2 控制帧字段建议

| 字段 | 类型 | 范围/示例 | 说明 |
|---|---|---|---|
| `type` | string | `control` | 消息类型 |
| `seq` | number | 递增整数 | 序号，用于判断丢包、重复和乱序 |
| `mode` | string | `stop` / `manual` / `cruise` / `debug` | 工作模式 |
| `throttle` | number | 0~100 | 前进强度 |
| `yaw` | number | -100~100 | 左右转向，负值可定义为左转 |
| `pitch` | number | -100~100 | 上浮/下潜倾向 |
| `depth_target` | number | cm 或 m | 目标深度，可选 |
| `tail_amp` | number | 0~100 | 摆尾幅度，可选 |
| `tail_freq` | number | 0~5Hz | 摆尾频率，可选 |
| `trim` | number | -20~20 | 舵机中位微调，可选 |
| `emergency_stop` | boolean | true/false | 急停标志 |
| `timestamp` | number | 毫秒时间戳 | 可选，用于延迟估计 |

### 6.3 控制帧示例

前进：

```json
{
  "type": "control",
  "seq": 101,
  "mode": "manual",
  "throttle": 60,
  "yaw": 0,
  "pitch": 0,
  "emergency_stop": false
}
```

左转：

```json
{
  "type": "control",
  "seq": 102,
  "mode": "manual",
  "throttle": 50,
  "yaw": -40,
  "pitch": 0,
  "emergency_stop": false
}
```

停止/急停：

```json
{
  "type": "control",
  "seq": 103,
  "mode": "stop",
  "throttle": 0,
  "yaw": 0,
  "pitch": 0,
  "emergency_stop": true
}
```

参数配置：

```json
{
  "type": "config",
  "seq": 201,
  "tail_amp_limit": 80,
  "tail_freq_limit": 3.5,
  "servo_trim_tail": 0,
  "servo_trim_left_fin": -2,
  "servo_trim_right_fin": 2
}
```

心跳：

```json
{
  "type": "heartbeat",
  "seq": 301,
  "timestamp": 12345678
}
```

## 7. ESP32 回传给上位机的 JSON 数据帧

### 7.1 状态帧字段建议

| 字段 | 类型 | 说明 |
|---|---|---|
| `type` | string | `telemetry` |
| `seq` | number | ESP32 回传序号 |
| `battery` | number | 电池电压 |
| `rssi` | number | WiFi 信号强度 |
| `mode` | string | 当前工作模式 |
| `depth` | number | 当前深度 |
| `roll` | number | 横滚角 |
| `pitch` | number | 俯仰角 |
| `yaw` | number | 航向角 |
| `stm32_online` | boolean | STM32 是否在线 |
| `error` | number | 错误码 |

示例：

```json
{
  "type": "telemetry",
  "seq": 501,
  "battery": 7.4,
  "rssi": -58,
  "mode": "manual",
  "depth": 0.35,
  "roll": 1.2,
  "pitch": -2.8,
  "yaw": 86.5,
  "stm32_online": true,
  "error": 0
}
```

### 7.2 ACK 应答帧

关键指令需要 ESP32 回 ACK 给网页，例如急停、参数配置、模式切换。

```json
{
  "type": "ack",
  "seq": 201,
  "ok": true,
  "message": "config applied"
}
```

错误应答：

```json
{
  "type": "ack",
  "seq": 201,
  "ok": false,
  "error": 3,
  "message": "invalid parameter"
}
```

## 8. 上位机应发送什么控制信号

### 8.1 推荐发送“运动意图”

不建议上位机直接发送每个舵机的角度。更推荐发送高层控制量：

| 控制量 | 范围示例 | 含义 |
|---|---:|---|
| `mode` | `stop` / `manual` / `cruise` | 控制模式 |
| `throttle` | 0~100 | 前进强度 |
| `yaw` | -100~100 | 左右转向 |
| `pitch` | -100~100 | 上浮/下潜倾向 |
| `depth_target` | cm 或 m | 目标深度 |
| `tail_amp` | 0~100 | 摆尾幅度 |
| `tail_freq` | 0~5Hz | 摆尾频率 |
| `emergency_stop` | true/false | 急停 |

### 8.2 STM32 侧运动映射思路

STM32 可以将高层控制量转换为舵机和电机输出，例如：

```text
tail_freq = map(throttle, 0~100, 0~MAX_FREQ)
tail_amp  = map(throttle, 0~100, 0~MAX_AMP)
yaw_offset = map(yaw, -100~100, -MAX_YAW_OFFSET~MAX_YAW_OFFSET)

tail_angle = center + sin(time * tail_freq) * tail_amp + yaw_offset
left_fin_angle  = left_center  + pitch_control + roll_correction
right_fin_angle = right_center + pitch_control - roll_correction
```

这样做的好处：

| 好处 | 说明 |
|---|---|
| 控制更稳定 | 上位机不需要关心舵机实时波形 |
| 实时性更好 | STM32 本地生成 PWM 和控制曲线 |
| 更易扩展 | 后续可加入深度闭环、姿态闭环、巡航模式 |
| 更安全 | 通信短时丢包时 STM32 可继续安全处理 |

## 9. ESP32 与 STM32 的 UART 协议

### 9.1 初期方案

初期调试时，为了快速跑通链路，可以使用：

| 方案 | 特点 |
|---|---|
| 固定字符串 | 最容易验证 UART 是否连通 |
| 简单 JSON | 易调试，但占用 RAM 和带宽略多 |
| 简化二进制帧 | 较正式，开发量中等 |

建议路线：

```text
固定字符串互发 → 简单帧格式 → 加校验 → 加 ACK → 加超时重发
```

### 9.2 正式二进制 UART 帧格式

推荐帧格式：

| 字段 | 长度 | 说明 |
|---|---:|---|
| 帧头 | 2 字节 | `0xAA 0x55` |
| 协议版本 | 1 字节 | 例如 `0x01` |
| 设备 ID | 1 字节 | 区分目标设备或广播 |
| 消息类型 | 1 字节 | 控制、状态、ACK、错误 |
| 序号 | 2 字节 | 用于去重、重发、乱序检测 |
| 命令字 | 1 字节 | 具体控制命令 |
| 数据长度 | 1 字节 | payload 长度 |
| 数据区 | N 字节 | 控制参数或状态数据 |
| CRC16 | 2 字节 | 校验 |

示例：

```text
AA 55 01 01 01 04 00 10 04 3C EC 00 1E CRC_L CRC_H
```

可解释为：

| 字段 | 含义 |
|---|---|
| `AA 55` | 帧头 |
| `01` | 协议版本 |
| `01` | 设备 ID |
| `01` | 控制消息 |
| `04 00` | 序号 4 |
| `10` | 手动控制命令 |
| `04` | payload 长度 4 |
| `3C EC 00 1E` | throttle=60，yaw=-20，pitch=0，depth=30 |
| `CRC_L CRC_H` | CRC16 |

### 9.3 命令字建议

| 命令字 | 方向 | 作用 |
|---|---|---|
| `0x01` | ESP32 → STM32 | 心跳 |
| `0x02` | STM32 → ESP32 | 心跳响应 |
| `0x10` | ESP32 → STM32 | 手动控制：throttle/yaw/pitch |
| `0x11` | ESP32 → STM32 | 设置单个舵机角度，调试用 |
| `0x12` | ESP32 → STM32 | 设置多个舵机动作，调试用 |
| `0x13` | ESP32 → STM32 | 设置运动模式 |
| `0x20` | ESP32 → STM32 | 请求传感器数据 |
| `0x30` | STM32 → ESP32 | 返回传感器数据 |
| `0x31` | STM32 → ESP32 | 返回执行机构状态 |
| `0x40` | STM32 → ESP32 | 错误码/异常状态 |
| `0x7F` | 双向 | ACK 应答 |

## 10. 串口接收解析逻辑

UART 接收不是天然按完整包到达，不能假设一次 receive 就是一整帧。应使用接收缓冲区和状态机解析。

状态机逻辑：

```text
状态 = 等待帧头1

每收到一个字节:
    如果状态是等待帧头1:
        如果字节 == 0xAA:
            状态 = 等待帧头2

    如果状态是等待帧头2:
        如果字节 == 0x55:
            状态 = 读取协议版本
        否则:
            回到等待帧头1

    读取协议版本
    读取设备ID
    读取消息类型
    读取序号
    读取命令字
    读取长度
    根据长度读取 payload
    读取 CRC

    如果 CRC 正确:
        生成完整消息
        交给命令处理函数
    否则:
        丢弃该帧，可记录错误

    回到等待帧头1
```

STM32 侧建议使用 UART 中断或 DMA + 环形缓冲区，避免阻塞主循环和舵机 PWM 控制。

## 11. 软件可靠性与抗丢包设计

无线信号损失无法完全靠软件解决，但可以通过协议和控制策略降低影响。

### 11.1 可靠性机制

| 机制 | 作用 |
|---|---|
| `seq` 序号 | 判断丢包、重复包和乱序包 |
| CRC 校验 | 判断数据是否损坏 |
| ACK 应答 | 确认关键指令已收到 |
| 超时重发 | 关键指令丢失后重新发送 |
| 心跳包 | 判断连接是否仍然存在 |
| 看门狗 | 程序异常时自动复位或进入安全状态 |
| 状态回传 | 上位机实时知道鱼当前状态 |
| 限频发送 | 避免指令过密造成拥塞 |
| 参数限幅 | 防止异常指令导致舵机过冲 |
| 指令平滑 | 避免控制量突变引起机械冲击 |
| 超时停机 | 上位机失联时自动停止或降速 |

### 11.2 指令分类处理

| 指令类型 | 示例 | 处理方式 |
|---|---|---|
| 关键指令 | 急停、模式切换、校准 | 必须 ACK，失败重发 |
| 实时遥控 | throttle、yaw、pitch | 不重发旧包，只执行最新帧 |
| 参数配置 | PID、舵机中位、限幅 | 必须 ACK，必要时保存到 Flash |
| 状态数据 | 姿态、电压、深度 | 可丢旧帧，只显示最新值 |

实时遥控指令不要频繁重发旧包，因为旧控制量可能已经过时。更推荐：

```text
如果收到更新的 seq:
    执行新控制帧
如果收到旧 seq:
    丢弃
```

### 11.3 超时保护建议

| 层级 | 超时条件 | 建议动作 |
|---|---|---|
| 上位机 ↔ ESP32 | 300ms 未收到控制帧 | 保持短时当前状态或逐渐降速 |
| 上位机 ↔ ESP32 | 1s 未收到控制帧 | ESP32 发送 STOP 给 STM32 |
| ESP32 ↔ STM32 | 500ms 未收到心跳 | STM32 停止舵机或进入安全姿态 |
| 任意层级 | 收到急停 | 立即停止推进并进入安全状态 |

伪代码：

```c
if now - last_control_time > 1000ms:
    mode = STOP
    send_stop_to_stm32()
```

## 12. ESP32 侧软件结构

| 模块 | 功能 |
|---|---|
| WiFi AP 模块 | 创建热点 |
| WebServer 模块 | 提供控制网页 |
| WebSocket 模块 | 接收上位机控制数据，推送状态 |
| JSON 解析模块 | 解析控制帧、配置帧、心跳帧 |
| UART 协议模块 | 与 STM32 互发二进制帧 |
| 状态管理模块 | 保存当前模式、控制量、连接状态 |
| 安全保护模块 | 超时停机、参数限幅、错误处理 |

ESP32 伪代码：

```c
初始化 WiFi AP
启动 WebServer
启动 WebSocket
初始化 UART
初始化心跳定时器

当 WebSocket 收到上位机消息:
    解析 JSON
    检查 seq 是否过旧
    检查参数范围

    如果 type == "control":
        更新 last_control_time

        如果 emergency_stop == true:
            生成 STOP 指令
            通过 UART 发送给 STM32
            返回 ACK 给上位机
        否则:
            将 throttle/yaw/pitch 转为 STM32 控制帧
            通过 UART 发送给 STM32

    如果 type == "config":
        检查参数合法性
        发送配置帧给 STM32
        等待 STM32 ACK
        回复上位机 ACK

主循环:
    如果超过一段时间没有收到控制帧:
        发送 STOP 或降速指令给 STM32

    如果 UART 收到 STM32 状态:
        解析状态帧
        打包成 telemetry JSON
        通过 WebSocket 推送给上位机

    定时发送 heartbeat 给 STM32
```

## 13. STM32 侧软件结构

| 模块 | 功能 |
|---|---|
| UART 接收模块 | 中断或 DMA 接收 ESP32 指令 |
| 协议解析模块 | 帧头、长度、CRC、命令解析 |
| 指令执行模块 | 根据命令更新模式和目标控制量 |
| 舵机 PWM 模块 | 控制尾部、背鳍、腹鳍舵机 |
| 电机 PWM 模块 | 控制推进电机或尾鳍驱动机构 |
| 传感器模块 | IMU、深度传感器、漏水检测、电压检测 |
| 控制算法模块 | 摆尾曲线、转向、上浮下潜、姿态修正 |
| 安全保护模块 | 通信超时、低电压、漏水、执行机构异常 |

STM32 伪代码：

```c
初始化 UART
初始化 PWM 舵机
初始化电机驱动
初始化传感器
初始化定时器

主循环:
    如果 UART 接收到新数据:
        写入环形缓冲区
        尝试解析完整帧

        如果 CRC 正确:
            根据命令字执行动作

            如果是手动控制:
                更新 throttle/yaw/pitch
                返回 ACK 或状态

            如果是设置运动模式:
                修改当前控制模式
                返回 ACK

            如果是请求传感器:
                读取传感器数据
                打包后发送给 ESP32

        如果 CRC 错误:
            丢弃该帧
            可返回错误码

    根据当前运动模式:
        计算舵机目标角度
        更新 PWM 输出

    定时采集传感器
    定时上报状态

    如果 ESP32 心跳超时:
        进入 STOP 或安全姿态
```

## 14. 上位机网页侧软件结构

网页可以由 ESP32 内置 WebServer 提供。用户不需要安装 App，只需浏览器访问 `http://192.168.4.1`。

### 14.1 推荐界面控件

| 控件 | 功能 |
|---|---|
| 启动/停止按钮 | 切换运行状态 |
| 急停按钮 | 立即停止所有推进动作 |
| 虚拟摇杆 | 控制 throttle 和 yaw |
| 上浮/下潜按钮或滑条 | 控制 pitch 或 depth_target |
| 模式选择 | stop、manual、cruise、debug |
| 速度滑条 | 控制推进强度或尾摆频率 |
| 幅值滑条 | 控制尾摆幅度 |
| 状态显示 | 电压、深度、姿态、WiFi RSSI、错误码 |

### 14.2 网页伪代码

```js
连接 WebSocket("ws://192.168.4.1/ws")

每 50ms:
    读取摇杆、按钮、滑条状态

    生成 JSON:
        type = "control"
        seq = seq + 1
        mode = 当前模式
        throttle = 油门值
        yaw = 转向值
        pitch = 上浮下潜值
        emergency_stop = 是否急停

    发送 JSON 给 ESP32

收到 ESP32 回传:
    如果 type == "telemetry":
        更新电池电压
        更新深度
        更新姿态
        更新信号强度
        显示错误码

如果 WebSocket 断开:
    页面提示失联
    尝试重连
```

### 14.3 推荐通信频率

| 内容 | 推荐频率 |
|---|---:|
| 手动遥控指令 | 20Hz 左右 |
| 姿态/深度状态回传 | 5~20Hz |
| 电池电压 | 1~2Hz |
| 心跳包 | 2~5Hz |
| 参数配置 | 按需发送 |

## 15. BLE 的定位

ESP32-S3 支持 BLE，但不建议将 BLE 作为第一阶段唯一实时遥控方案。BLE 更适合备用控制和参数配置。

| BLE 用途 | 是否推荐 |
|---|---|
| 参数配置 | 推荐 |
| 低频控制 | 可用 |
| 应急停止 | 可作为备用 |
| 高频实时遥控 | 不优先 |
| 大量状态回传 | 不优先 |

BLE 可设计为 GATT 服务：

| 特征值 | 作用 |
|---|---|
| Control Characteristic | 上位机写入控制命令 |
| Telemetry Characteristic | ESP32 notify 回传状态 |
| Config Characteristic | 配置 PID、舵机中位、限幅等参数 |

## 16. 分阶段实现路线

| 阶段 | 目标 | 验收标准 |
|---|---|---|
| 1 | ESP32 开启 WiFi AP | 手机/电脑能连接热点 |
| 2 | ESP32 提供网页 | 浏览器能打开 `http://192.168.4.1` |
| 3 | 网页连接 WebSocket | 页面能与 ESP32 实时收发文本 |
| 4 | 网页发送 JSON 控制帧 | ESP32 串口日志能看到解析后的控制量 |
| 5 | ESP32 与 STM32 UART 互发 | 两块芯片能互发固定字符串 |
| 6 | UART 简单协议 | STM32 能识别 throttle/yaw/pitch |
| 7 | STM32 控制舵机/电机 | 舵机和推进机构能按指令动作 |
| 8 | STM32 回传状态 | 网页能显示电压、姿态、深度、错误码 |
| 9 | 加入 seq、CRC、ACK | 能处理丢包、错误帧、关键指令确认 |
| 10 | 加入超时停机 | 上位机断开后鱼能自动停止 |
| 11 | 优化控制算法 | 实现稳定前进、转向、上浮、下潜 |
| 12 | 水池测试 | 验证通信距离、控制延迟、防水和安全保护 |

## 17. 关键风险与应对

| 风险 | 影响 | 应对 |
|---|---|---|
| 2.4GHz 水下衰减 | 控制距离短、信号丢失 | 天线靠近水面或露出水面，优先 AP 模式，加入失联保护 |
| 馈线损耗 | 信号质量下降 | 使用合格 50Ω 同轴馈线，尽量缩短馈线 |
| 舵机/电机干扰 | ESP32 重启、UART 错误 | 电源分级、共地、加电容、走线远离大电流回路 |
| UART 丢包 | 指令异常 | 帧头、长度、CRC、状态机解析 |
| WiFi 临时断连 | 鱼失控 | 心跳、超时降速、1s 停机 |
| JSON 字段异常 | 控制量越界 | ESP32 参数限幅，STM32 再次限幅 |
| STM32 卡死 | 执行机构持续动作 | STM32 看门狗，ESP32 心跳监测 |
| ESP32 卡死 | 无线失联 | ESP32 看门狗，STM32 通信超时进入 STOP |
| 上位机误操作 | 急剧转向或推进 | 控制量平滑、最大变化率限制、急停优先 |

## 18. 最终推荐方案

| 设计项 | 推荐选择 |
|---|---|
| 无线链路 | ESP32-S3 WiFi AP |
| 上位机形态 | 浏览器网页，不先写原生 App |
| 网络通信 | WebSocket |
| 上位机数据格式 | JSON |
| 芯片间通信 | UART |
| UART 正式格式 | 二进制帧 + CRC16 |
| 控制信号 | throttle、yaw、pitch、mode、emergency_stop |
| 反馈信号 | battery、rssi、depth、roll、pitch、yaw、error |
| 安全机制 | 心跳、超时停机、ACK、参数限幅、看门狗 |
| 后续扩展 | BLE 参数配置、UDP 低延迟控制、原生 App |

一句话总结：

> 第一阶段采用 **ESP32-S3 WiFi AP + 浏览器网页 + WebSocket + JSON** 作为上位机通信方式；ESP32-S3 负责无线通信和协议转换，STM32F103C8T6 负责实时运动控制。上位机发送“想怎么游”的高层控制意图，STM32 再转换为具体舵机、电机和传感器闭环控制。
